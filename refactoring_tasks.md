# 今後のリファクタリングタスク（Supabase/API・スキーマ変更関連）

Astro側のフロントエンドやAPIの基本的なリファクタリングは完了しましたが、以下の項目はデータベース（Supabase）のスキーマ変更、RLSポリシーの変更、または影響範囲が広く動作確認を要するAPIの根本的な変更が含まれるため、手動タスクとして起票します。

## 対象箇所と提案

### 1. 予約時間の重複防止機能の強化（対象：Supabase RLS / DB トリガー）
- **緊急度**: 中
- **現状**: 現在はPostgreSQLの `EXCLUDE` 制約や、`start_time` と `end_time` に対する制約で時間の重複を防いでいますが、`isExclusionViolation` で拾うためのエラーハンドリングに依存しています。
- **提案**: RLSポリシー内で他の予約と重複しないかをチェックする、もしくはSupabase Edge Functions（またはAPI層）でDB挿入前に予約の重複をトランザクション内でチェックし、より明確なエラーレスポンスを返すアーキテクチャへの移行を検討します。

### 2. 過去の予約ロック機能の改善（対象：Supabase RLS）
- **緊急度**: 中
- **現状**: 予約の変更・削除について、フロントエンド側での時間ロック（1時間経過後）と、API側のDBエラー（トリガー）を組み合わせて防いでいます。
- **提案**: データの完全性を担保するため、Supabase の RLS（Row Level Security）ポリシー内で `now() > start_time + interval '1 hour'` といった条件を追加し、セキュリティ層で直接UPDATEやDELETEを弾く設計にすることで、より堅牢なシステムになります。

### 3. APIルートの認証・バリデーション共通化（対象：`src/pages/api/*`）
- **緊急度**: 低
- **現状**: 各APIエンドポイント（`reserve.ts`, `facilities.ts`, `members.ts` など）で、セッションの取得や権限チェック（`if (!session || !member) ...`）のコードが重複しています。
- **提案**: Astroのミドルウェア（`src/middleware.ts`）でAPI配下のルートに対する認証をより厳密に行うか、`api-utils.ts` に共通のバリデーションラッパー関数（HOCのようなもの）を作成して処理を共通化することで、コードの重複を減らし、セキュアに保つことができます。

### 4. データベースのインデックス最適化（対象：Supabase Schema）
- **緊急度**: 低
- **現状**: 予約データの取得が多く、特に `start_time` や `facility_id` での絞り込みが頻繁に行われます。
- **提案**: `reservations` テーブルの `start_time`, `end_time`, `facility_id` にインデックス（B-Tree など）を適切に張ることで、カレンダーやタイムライン表示時のデータ取得パフォーマンスが向上します。現状のレコード数では問題ありませんが、今後のデータ増加に備えた最適化として検討してください。
